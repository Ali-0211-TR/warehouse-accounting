## 2. Критические ошибки в расчётах

### 🔴 2.1 Ошибка в формуле `totalSum` (backend)

**Файл:** `src-tauri/src/infrastructure/repositories/order/order_fetching.rs`

**Текущая формула:**
```
total_sum = -income_sum + outcome_sum + sale_sum + sale_dispenser_sum - returns_sum
total_tax = -income_tax + outcome_tax + sale_tax + sale_dispenser_tax - returns_tax
```

**Проблема:** Семантика формулы запутана. `Outcome` (возврат поставщику) считается как **приход денег** (`+`), а `Returns` (возврат от покупателя) считается как **расход** (`-`). Это **верно с точки зрения денежного потока**, но:

1. **На фронтенде** в `SummaryReportPage.tsx` используется та же формула дублированно:
   ```tsx
   const totalIncoming = saleSum + saleTax + saleDispenserSum + saleDispenserTax + outcomeSum + outcomeTax;
   const totalOutgoing = incomeSum + incomeTax + returnsSum + returnsTax;
   const netFlow = meta.totalSum; // из бекенда
   ```
   Если бекенд изменит формулу — фронтенд сломается, потому что `netFlow` вычисляется **и на бекенде, и на фронтенде** параллельно.

2. **Нет кросс-валидации:** `totalIncoming - totalOutgoing` на фронтенде может не совпадать с `meta.totalSum` из бекенда из-за разницы в округлении или включении скидок.

**Рекомендация:**
- Убрать дублирование расчётов — бекенд должен возвращать `totalIncoming` и `totalOutgoing` как готовые поля
- Добавить assertion-проверку: `totalIncoming - totalOutgoing === netFlow`
- Вынести формулу в единое место (shared type/constant)

### 🔴 2.2 Баланс товара считается без фильтрации удалённых ордеров

**Файл:** `src-tauri/src/infrastructure/repositories/product_repository.rs`

**Текущий SQL:**
```sql
UPDATE products SET balance = (
    SELECT SUM(
        CASE WHEN orders.order_type IN ('Income', 'Returns')
             THEN order_items.count
             ELSE order_items.count * -1
        END
    )
    FROM order_items
    INNER JOIN orders ON order_items.order_id = orders.id
    WHERE order_items.product_id = products.id
)
```

**Проблемы:**
1. **Нет фильтра `deleted_at IS NULL`** — удалённые ордеры всё ещё учитываются в балансе
2. **Нет фильтра по `d_move`** — ордеры без подтверждённой даты движения уже влияют на баланс
3. **NULL balance** — если нет ни одного order_item, баланс становится `NULL`, а не `0`

**Рекомендация:**
```sql
UPDATE products SET balance = COALESCE((
    SELECT SUM(...)
    FROM order_items
    INNER JOIN orders ON order_items.order_id = orders.id
    WHERE order_items.product_id = products.id
      AND orders.deleted_at IS NULL
      AND order_items.deleted_at IS NULL
), 0)
```

### 🔴 2.3 Товары из SaleDispenser-ордеров неверно классифицируются

**Файл:** `src-tauri/src/infrastructure/repositories/order/order_fetching.rs`

**Текущая логика:**
- Query 3 находит товары в `SaleDispenser` ордерах, у которых нет `fueling_order`
- Их суммы **добавляются к `sale_sum`/`sale_tax`**

**Проблема:** Если оператор при заправке добавляет в ордер сопутствующий товар (масло, антифриз и т.п.), эти товары попадают в "обычные продажи" (`Sale`), хотя фактически являются частью ордера на ТРК. Это:
- Искажает статистику **обычных продаж** (они завышены)
- Скрывает **сопутствующие продажи при заправке** — важный бизнес-показатель

**Рекомендация:**
- Ввести отдельную категорию `saleDispenserGoods` в `totalsByType`
- Или добавить флаг `is_fuel_related: boolean` в `OrderItemEntity`

---

### 🟡 2.4 Скидки не учитываются в расчёте `totalSum`

**Текущая формула:** `totalSum` учитывает только `sum` и `tax`, но не `discount`.

В `totalsByType` скидки есть (`saleDiscount`, `incomeDiscount` и т.д.), но:
- В `totalIncoming`/`totalOutgoing` на фронтенде скидки **не вычитаются**
- `enhancedNetWorth = netFlow + totalSaleProduct` — скидки игнорируются

**Рекомендация:**
- Определить, что означает `totalSum`: сумма до скидки или после
- Унифицировать формулу: `netAmount = sum - discount + tax` или `sum + tax` (если скидка уже учтена в `cost`)
- Задокументировать формулу в коде

---

### 🟡 2.5 `enhancedNetWorth` — спорный показатель

```tsx
const enhancedNetWorth = netFlow + totalSaleProduct;
```

Это смешивает **реализованный денежный поток** с **нереализованной стоимостью инвентаря**. Проблемы:
- Завышает финансовое положение (инвентарь ещё не продан)
- Не учитывает себестоимость (`totalIncomeProduct`)
- Если инвентарь пропадёт/испортится — показатель будет фиктивным

**Рекомендация:**
- Показывать отдельно: «Денежный поток», «Стоимость инвентаря», «Чистая позиция»
- Не объединять в один "Business Value"

## 3. Проблемы прозрачности и посменного учёта

### 🔴 3.1 Отчёт по сменам не содержит финансовых данных

**Файл:** `report-shift/shift-report-page.tsx`

Текущий отчёт по сменам — это просто **список смен** (дата открытия, закрытия, оператор). **Нет:**
- Суммы продаж за смену
- Суммы возвратов за смену
- Суммы закупок за смену
- Количества ордеров за смену
- Выручки за смену
- Расхода топлива за смену
- Сравнения план/факт

**Рекомендация:**
Полностью переработать `ShiftReportPage` — сделать **детальный посменный отчёт** (см. раздел 5.1).

---

### 🔴 3.2 Движение товаров не привязано к смене по умолчанию

**Файл:** `report-movements/MovementsReportPage.tsx`

Текущая логика при загрузке:
```tsx
const shift = useShiftStore.getState().currentShift;
if (shift) {
    loadData({ filters: { d_move: [shift.d_open, dClose] } });
} else {
    loadData(); // ← загружает ВСЕ данные без фильтра
}
```

**Проблемы:**
- Если нет открытой смены → показываются **все ордеры за всё время** → огромный объём данных
- Нет индикации периода, за который показаны данные
- `ShiftSelector` в header не связан с фильтрами основного компонента через единый state

**Рекомендация:**
- По умолчанию показывать данные за **последнюю смену** или **сегодня**
- Добавить видимый индикатор текущего фильтра периода
- Связать `ShiftSelector` с остальными фильтрами через единый store

### 🔴 3.3 Сводный отчёт (Summary) не фильтруется по сменам

**Файл:** `report-summary/SummaryReportPage.tsx`

```tsx
useEffect(() => {
    loadData();     // ← загружает ВСЕ ордеры
```

- Сравнить разные периоды

**Рекомендация:**
- Добавить `DateFilterSelector` / `ShiftSelector`
- По умолчанию — текущая смена или сегодня
- Добавить кнопку "Все данные" для полной сводки

### 🟡 3.4 Нет сквозного учёта: "Начальный остаток → Приход → Продажа → Возврат → Расход → Конечный остаток"

Ни в одном отчёте нет классической формулы товародвижения:

```
Конечный остаток = Начальный остаток + Приход - Расход - Продажа + Возвраты
```

**Рекомендация:**
- Создать отчёт «Товародвижение по товару» с формулой
- Показывать начальный остаток на начало смены/периода
- Показывать каждую операцию
- Итоговый остаток должен совпадать с расчётным балансом

## 4. Проблемы UX/UI

### 🔴 4.1 Сводный отчёт перегружен (1531 строк, 3 графика, 20+ карточек)

**Файл:** `SummaryReportPage.tsx` — **1531 строк** кода в одном компоненте.

**Проблемы:**
- Информационная перегрузка: Revenue Sources, Expense Sources, Quick Stats, Financial Health, Detailed Breakdown, Financial Flow Analysis, Value Distribution, Performance Analysis, Business Intelligence Summary — всё на одной странице
- Дублирование: "Detailed Financial Breakdown" показывается **дважды** (один раз с BadgeS, второй раз без)
- Графики (`recharts`) плохо работают на мобильных
- Нет приоритезации — пользователь не понимает, что важнее
- "Business Intelligence Summary" с показателями типа "Liquidity Ratio" и "Inventory Dependency" — слишком сложные для оператора АЗС

**Рекомендация:**
- Разбить на подкомпоненты и вкладки (Tabs):
  - **Обзор** — ключевые метрики (выручка, расходы, прибыль)
  - **Детали** — разбивка по типам операций
  - **Аналитика** — графики и BI-метрики
- Убрать дублирование блоков
- Адаптировать под мобильную версию
- Вынести сложные метрики в отдельный "экспертный" режим

---

### 🔴 4.2 Отчёт движения топлива слишком сложный (1235 строк)

**Файл:** `FuelMovementsReportPage.tsx` — **1235 строк**, включает:
- Чтение данных с ТРК (real-time)
- Данные из БД (summary)
- Фильтрация по дисперсерам/танкам/сменам
- Группировка по танкам
- Экспорт в CSV
- Отображение таблицы + карточек для мобильных
- Диалог фильтров с формой

**Проблемы:**
- Компонент делает **слишком много** — нарушение принципа единственной ответственности
- 3 разных источника данных (real-time, shift, DB) отображаются в одной таблице — пользователь не понимает разницу
- `shiftVolume/shiftAmount` — данные **из электроники ТРК** (счётчик смены), а не из БД
- `foVolume/foAmount` — данные **из БД** (заправочные ордеры)
- Разница между ними может быть из-за: незавершённых заправок, потерь, ошибок связи — это **не объясняется** пользователю

**Рекомендация:**
- Разделить на 3 подкомпонента: «Real-time ТРК», «Данные по сменам», «Сводка по БД»
- Добавить **индикатор расхождения** между данными ТРК и БД (подозрение на потери/ошибки)
- Вынести логику в отдельные хуки

## 5. Недостающие отчёты и функциональность

### 5.1 🆕 Детальный посменный отчёт (Shift Detail Report)

**Текущее:** Просто список смен.
**Нужно:** При клике на смену → полный финансовый отчёт за смену.

**Содержание:**
- 📊 **Заголовок:** Смена №X, Оператор, Дата/Время открытия-закрытия, Длительность
- 💰 **Финансы:**
  - Выручка за смену (продажи + заправки)
  - Закупки за смену
  - Возвраты за смену (клиентские + поставщику)
  - Чистый результат смены
- ⛽ **Топливо:**
  - Продано литров по каждому виду топлива
  - Количество заправок
  - Средний чек заправки
  - Данные счётчиков ТРК (начало/конец смены)
- 🛒 **Товары:**
  - Продано единиц по каждому товару
  - Начальный/конечный остаток по каждому товару
  - Недостачи/излишки
- 📋 **Список ордеров:** Все ордеры за смену
- 🖨️ **Печать:** Готовый к печати чек/отчёт закрытия смены

---

### 5.2 🆕 Отчёт по товару (Product Movement Report)

**Текущее:** Только текущий остаток.
**Нужно:** Полная история движения конкретного товара.

**Содержание:**
- Выбор товара из списка
- Выбор периода
- Таблица:
  | Дата | Операция | Кол-во | Цена | Сумма | Остаток |
  |------|----------|--------|------|-------|---------|
  | 01.02 | Приход | +100 | 5000 | 500,000 | 100 |
  | 02.02 | Продажа | -10 | 7000 | 70,000 | 90 |
  | ... | ... | ... | ... | ... | ... |

- Итого: начальный остаток, приход, расход, продажа, возвраты, конечный остаток
- Проверка: `расчётный остаток == фактический остаток`

---

### 5.3 🆕 Отчёт по клиенту/контракту

**Текущее:** Нет.
**Нужно:** Все операции по конкретному клиенту или контракту.

- Сумма покупок клиента за период
- Задолженность по контракту
- Остаток лимита
- Список операций

---

### 5.4 🆕 Сверочный отчёт (Reconciliation Report)

**Текущее:** Нет.
**Нужно:** Выявление расхождений.

- **Топливо:** Данные ТРК vs. Данные БД → расхождение
- **Товары:** Расчётный баланс vs. Фактический (инвентаризация)
- **Касса:** Сумма наличных продаж vs. Фактическая касса
- Визуальная индикация: 🟢 совпадает, 🟡 небольшое расхождение, 🔴 критическое расхождение

---

### 5.5 🆕 Сравнительный отчёт по периодам

**Текущее:** Нет.
**Нужно:** Сравнение двух периодов (смен, дней, недель, месяцев).

- Таблица с двумя колонками: Период 1 | Период 2 | Разница | %
- По каждому виду топлива, товаров
- Тренды роста/падения

---

### 5.6 🆕 Дашборд (Dashboard)

**Текущее:** `SummaryReportPage` пытается быть дашбордом.
**Нужно:** Отдельная главная страница с ключевыми метриками.

- Остатки в танках (из tank monitoring)
- Последние операции
- Уведомления/алерты (низкий остаток, большая скидка, аномальная операция)
